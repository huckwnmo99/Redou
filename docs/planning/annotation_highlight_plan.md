# Annotation and Highlight Plan

이 문서는 이 앱의 `PDF reader + highlight + paragraph note` 기능을 대략적으로 정리한 초안이다.  
핵심 아이디어는 단순 형광펜이 아니라, **사용자 의미 체계를 가진 annotation 시스템**으로 설계하는 것이다.

---

## 1. 왜 필요한가

논문을 저장만 해두면 나중에 다시 찾을 때 "왜 중요했는지"가 남지 않는다.  
반대로 읽는 중에 문단 단위로 하이라이트와 메모를 남기면 아래 가치가 생긴다.

- 중요한 문단을 빠르게 다시 찾을 수 있다.
- 사용자가 왜 그 문단을 중요하게 봤는지 기록할 수 있다.
- 발표나 논문 작성 때 근거와 개인 해석을 함께 확인할 수 있다.
- 나중에 RAG를 붙일 때, 사용자가 표시한 문단을 우선적인 근거 후보로 사용할 수 있다.

즉 이 기능은 부가 기능이 아니라, 이 앱의 핵심 차별점 후보로 본다.

---

## 2. 핵심 원칙

### 2-1. 색이 아니라 카테고리가 핵심

하이라이트는 `색 = 의미`로 저장하지 않는다.  
정확히는 `카테고리 = 의미`, `색 = 시각 표현`으로 분리한다.

예:
- 노란색 = 중요 결과
- 빨간색 = 다시 볼 내용
- 파란색 = 방법 참고

이때 실제 저장은 "노란색"이 아니라 `중요 결과`라는 카테고리로 남는다.

이 방식의 장점:
- 사람마다 색 의미가 달라도 대응 가능
- 나중에 검색과 필터링이 쉬움
- 팀 공유 기능을 붙여도 사용자별 의미 체계를 유지할 수 있음
- 시각 표현을 바꾸더라도 데이터 의미는 유지됨

### 2-2. 기본 카테고리 + 사용자 정의 카테고리

카테고리는 두 층으로 구성하는 것이 좋다.

- 시스템 기본 카테고리
- 사용자 직접 생성 카테고리

이유:
- 처음 쓰는 사용자도 바로 시작할 수 있어야 한다.
- 동시에 사람마다 읽는 방식이 다르므로 커스터마이징이 가능해야 한다.

### 2-3. 하이라이트와 메모는 분리하되 연결 가능하게

- 하이라이트: 빠른 분류
- 메모: 해석, 생각, 의문, 활용 기록

즉:
- 어떤 문단은 색만 칠할 수 있다.
- 어떤 문단은 색칠 후 메모를 추가할 수 있다.
- 어떤 문단은 메모만 남길 수도 있다.

---

## 3. MVP에 포함할 annotation 기능

### 포함

- 앱 안에서 PDF 열기
- 텍스트 선택
- 선택 영역에 하이라이트 추가
- 하이라이트 카테고리 선택
- 선택적으로 메모 추가
- 하이라이트/메모를 문단 또는 텍스트 위치와 연결해 저장
- 하이라이트 목록 보기
- 카테고리별 필터링
- 하이라이트 클릭 시 원문 위치로 이동

### 제외

- 자유 드로잉
- PDF 위 손글씨
- 복잡한 협업 코멘트 스레드
- 동일 구간 다중 스타일 겹치기
- PDF 파일 자체에 영구 주석 삽입

이 항목들은 MVP 범위를 크게 늘리므로 제외하는 것이 맞다.

## 4. 기본 카테고리 초안

처음부터 최소한 아래 기본 카테고리는 내장하는 것이 좋다.

| 카테고리 | 의미 | 기본 색상 예시 |
|---|---|---|
| 중요 결과 | 핵심 결과, 주장, 수치 | 노랑 |
| 다시 볼 내용 | 아직 판단이 안 끝나 재확인이 필요한 부분 | 빨강 |
| 내 연구와 관련 | 내 주제와 직접 연결되는 부분 | 초록 |
| 방법 참고 | 실험 방법, 모델, 절차 참고용 | 파랑 |
| 발표 후보 | 발표 자료에 인용하거나 figure와 함께 쓸 후보 | 보라 대신 주황/앰버 계열 권장 |
| 의문점 | 이해가 안 되거나 검증이 필요한 부분 | 회색/슬레이트 |
| 주의/한계 | 한계점, 반례, 조건부 해석 | 분홍 또는 브릭 계열 |

주의:
- 색은 기본값일 뿐이고 사용자가 바꿀 수 있어야 한다.
- 카테고리 이름도 추후 사용자 정의로 추가 가능해야 한다.

---

## 5. 사용자 정의 카테고리 정책

사용자는 직접 하이라이트 카테고리를 만들 수 있어야 한다.

사용자 정의 가능 항목:
- 카테고리 이름
- 색상
- 설명
- 정렬 순서
- 활성/비활성 여부

추천 제한:
- 아이콘 선택은 MVP에서 없어도 된다.
- 계층형 카테고리는 MVP에서 필요 없다.
- 카테고리 수가 너무 많아지는 것을 막기 위해 적당한 관리 화면은 필요하다.

---

## 6. 하이라이트와 메모 저장 개념

정확한 스키마는 나중에 DB 설계에서 잡으면 되지만, 개념적으로는 아래처럼 분리하는 것이 좋다.

### 6-1. Highlight preset

의미:
- 사용자별 카테고리 정의

예시 필드:
- `preset_id`
- `user_id`
- `name`
- `color`
- `description`
- `is_default`
- `sort_order`
- `created_at`

### 6-2. Highlight

의미:
- 실제 논문 본문에 칠한 표시

예시 필드:
- `highlight_id`
- `paper_id`
- `user_id`
- `preset_id`
- `page`
- `section_name`
- `chunk_id`
- `selected_text`
- `start_anchor` (PDF 좌표/텍스트 anchor)
- `end_anchor` (PDF 좌표/텍스트 anchor)
- `created_at`

### 6-3. Note

의미:
- 하이라이트 또는 paper/section/chunk에 연결되는 메모

예시 필드:
- `note_id`
- `paper_id`
- `user_id`
- `highlight_id` nullable
- `section_name`
- `chunk_id`
- `page`
- `selected_text`
- `note_type`
- `note_text`
- `created_at`

핵심:
- `selected_text + page + chunk_id`를 같이 남겨야 나중에 위치 복구가 쉽다.
- anchor는 단순 문자 인덱스보다 `PDF 좌표 + 텍스트 anchor` 조합으로 가는 편이 안전하다.
- 하이라이트와 메모는 1:1 강제가 아니라 느슨하게 연결하는 편이 좋다.

## 7. local Supabase 관점

현재 local Supabase를 고려하고 있으므로 annotation 데이터도 구조화해서 저장하기 좋다.

권장 방향:
- `highlight_presets` 테이블
- `highlights` 테이블
- `notes` 테이블
- 필요하면 추후 `annotation_events` 또는 `activity_log` 추가

이 구조의 장점:
- paper, section, chunk, figure, notes와 쉽게 join 가능
- 카테고리별 검색이 쉬움
- 사용자별 preset 분리가 쉬움

---

## 8. 프론트엔드 UI 방향

### 8-1. PDF Reader 안에서의 상호작용

권장 흐름:

1. 사용자가 PDF에서 텍스트를 선택
2. 작은 popover 또는 floating toolbar 표시
3. 하이라이트 카테고리 선택
4. 즉시 색칠 저장
5. 필요하면 바로 메모 입력창 열기

이 흐름이 좋은 이유:
- 읽는 흐름을 크게 끊지 않는다.
- 메모를 강제하지 않아 빠르게 사용 가능하다.

### 8-2. 오른쪽 패널 활용

오른쪽 패널에는 아래 중 하나를 보여주는 방식이 좋다.

- 현재 문단의 메모
- 현재 페이지의 하이라이트 목록
- 현재 논문의 전체 annotation 목록
- 카테고리 필터

### 8-3. Overview와 Search에서의 활용

하이라이트 데이터는 PDF 화면에만 머무르면 안 된다.  
다른 화면에서도 써야 가치가 커진다.

활용 예:
- Paper Detail > Highlights 탭
- Search 결과에서 "중요 결과만 보기"
- 발표 준비 시 "발표 후보"만 필터링
- 내 연구 관련 하이라이트만 다시 보기

---

## 9. 검색과 RAG에서의 가치

하이라이트와 문단 메모는 나중에 retrieval 품질을 높이는 데 직접 쓸 수 있다.

예:
- `중요 결과` 카테고리가 달린 문단 우선 검색
- `발표 후보` 표시가 있는 figure/문단을 우선 추천
- `의문점` 메모가 있는 문단만 다시 모아보기
- 사용자가 작성한 메모를 기반으로 내 연구 관련 문단 재검색

즉 annotation 데이터는 단순 UI 정보가 아니라, 장기적으로 retrieval signal이 된다.

---

## 10. 제품 관점에서의 정의

이 기능은 단순 형광펜 기능이 아니라 아래처럼 정의하는 것이 맞다.

> 사용자가 논문을 읽는 동안 의미 있는 문단을 카테고리 기반으로 표시하고, 필요한 경우 메모를 남겨, 나중에 그 논문을 근거와 함께 다시 활용할 수 있게 만드는 annotation 시스템
