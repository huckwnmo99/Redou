# Windows Research Support Application 기획 결정 초안

이 문서는 구현 전에 제품 내용을 확정하기 위한 초안이다.  
현재 기준으로는 `논문 자동 정리 + 구조화 저장 + 메모 축적 + 근거 재검색 준비`를 핵심 목표로 두고 작성했다.

작성 원칙:
- 지금 당장 만들 것과 나중에 확장할 것을 분리한다.
- 구현 난이도가 높은 기능은 구조만 준비하고 MVP에서는 제외한다.
- 자동화 정확도는 100%를 전제하지 않고, 항상 수동 수정 흐름을 남긴다.

---

## 1. 제품 한 줄 정의

### 1-1. 제품 핵심 문장
이 앱은 무엇을 하는 앱인가?

`작성:`

> 연구자가 읽은 논문 PDF를 업로드하면 메타데이터, 구조화된 본문, figure, 메모를 함께 정리해 두고, 이후 발표 준비나 논문 작성 시 다시 찾고 재사용할 수 있게 만드는 Windows 기반 연구 정리 앱이다.

### 1-2. 이 앱이 아닌 것
혼동을 막기 위해, MVP 단계에서 이 앱이 **하지 않는 일**도 적는다.

`작성:`
- 완전 자동 논문 이해 엔진은 아니다.
- 대규모 외부 논문 추천 플랫폼은 아니다.
- ontology를 자동 구축하고 추론까지 수행하는 시스템은 아직 아니다.
- 팀 협업 중심 SaaS는 아니며, MVP는 개인 연구자 중심으로 본다.

### 1-3. 핵심 가치
이 제품이 반드시 만족해야 하는 가치 2~3개를 적는다.

`작성:`
- 실용성: 읽은 논문을 다시 찾고 발표/작성에 바로 써먹을 수 있어야 한다.
- 편리성: PDF를 넣으면 자동으로 최대한 정리되고 사용자는 수정과 메모에 집중할 수 있어야 한다.
- 근거 추적 가능성: 요약과 검색 결과가 항상 원문 위치와 연결되어 있어야 한다.
- 확장성: 나중에 RAG와 ontology를 붙일 수 있도록 저장 구조가 처음부터 구조화되어 있어야 한다.

## 2. 핵심 사용자와 사용 상황

### 2-1. 주요 사용자
누가 이 앱을 가장 자주 사용할 것인가?

`작성:`
- 주 사용자: 대학원생, 박사후연구원, 기업/기관 연구원
- 연구 분야: 공학/과학 계열의 실험 및 문헌 기반 연구
- 사용 숙련도: 논문 관리 도구를 써본 적은 있지만, 고급 세팅은 최소화하고 싶은 사용자
- 혼자 쓰는 앱인지 / 팀과 공유하는 앱인지: MVP는 개인 사용 중심, 구조는 향후 팀 공유 확장 가능하게 설계

### 2-2. 핵심 사용 상황 3개
실제로 이 앱을 여는 순간을 기준으로 적는다.

`작성:`
1. 새로 읽은 논문 PDF를 넣고 자동으로 정리 카드와 메모 공간을 만들고 싶을 때
2. 예전에 읽었던 논문에서 특정 결과, figure, 조건을 다시 찾고 싶을 때
3. 발표 자료나 논문 초안을 쓰면서 내 주장과 연결되는 근거 논문을 다시 확인하고 싶을 때

### 2-3. 지금 당장 해결하려는 가장 큰 문제
한 문장으로 적는다.

`작성:`

> 읽은 논문이 파일과 메모로 흩어져 있어서, 나중에 필요한 근거와 figure를 다시 찾는 데 시간이 너무 많이 든다.

### 3-1. MVP에 반드시 포함할 기능
`Must-have`만 적는다.

`작성:`
- [x] PDF 업로드
- [x] 메타데이터 자동 추출
- [x] 파일명 자동 정리
- [x] 논문 카드 생성
- [x] 메모 작성
- [x] 섹션 분리
- [x] chunk 저장
- [x] figure/caption 기본 정리
- [x] 제목/메타데이터/본문/메모 기반 검색
- [x] 태그
- [x] 사용자 폴더/컬렉션 분류
- [x] 읽음 상태 관리
- [x] 자동 추출 결과 수동 수정

### 3-2. MVP에서는 제외할 기능
지금은 하지 않기로 명시하는 항목.

`작성:`
- [x] ontology 자동 생성
- [x] 복잡한 인과관계 분석
- [x] 자동 knowledge graph 구축
- [x] 고도화된 claim-grounding 자동화
- [x] 다중 논문 자동 비교 리포트
- [x] 외부 논문 DB 실시간 연동
- [x] 팀 단위 실시간 협업 편집
- [x] figure 의미 이해 기반 고급 검색

### 3-3. 왜 제외하는가
나중으로 미루는 이유를 적는다.

`작성:`
- ingestion과 구조화 저장이 안정되지 않으면 고급 기능의 품질이 무너진다.
- ontology와 지식 그래프는 데이터 모델이 먼저 안정된 뒤 붙이는 편이 맞다.
- 협업 기능을 초기에 넣으면 권한, 동기화, 충돌 해결까지 범위가 크게 늘어난다.
- MVP에서는 "정리와 재검색" 문제를 푸는 데 집중해야 한다.

## 4. PDF 입력 범위

### 4-1. 지원 대상
어떤 PDF를 안정적으로 처리할지 먼저 정한다.

`작성:`
- 언어: 1차는 영어 논문 중심
- 학술 논문 형식: 저널 논문, 학회 논문, arXiv 형태의 born-digital PDF
- 1단 / 2단 레이아웃 지원 여부: 둘 다 지원
- 스캔 PDF 지원 여부: 지원
- OCR 지원 여부: MVP 포함
- supplementary PDF 포함 여부: 기본 업로드는 가능하되 메인 논문과 분리 저장
- 이미지 중심 PDF 처리 여부: OCR을 통해 제한적으로 지원

### 4-2. 지원하지 않는 입력

`작성:`
- 논문 형식이 아닌 슬라이드, 포스터, 브로셔 PDF
- 표준 레이아웃이 크게 무너진 문서
- OCR로도 텍스트 복구가 거의 되지 않는 저품질 스캔본

### 4-3. 실패 시 사용자 경험
추출 실패가 났을 때 어떻게 보여줄지 적는다.

`작성:`
- 메타데이터 추출 실패 시: 제목/저자/연도 입력 칸을 비운 상태로 카드 생성 후 수동 입력 유도
- 섹션 분리 실패 시: 전체 본문 텍스트를 단일 본문으로 저장하고 "section parsing failed" 상태 표시
- DOI 추출 실패 시: DOI 없음으로 저장하되 사용자가 직접 입력 가능
- figure 추출 실패 시: 본문/메타데이터 저장은 계속 진행하고 figure 영역만 빈 상태로 둠
- OCR 실패 시: 스캔 PDF 상태를 표시하고 수동 메타데이터 입력 및 부분 텍스트 보정 유도

## 5. 파일 관리 정책

### 5-1. 파일 저장 방식
원본 PDF를 어떻게 다룰지 결정한다.

`작성:`
- 앱이 파일을 복사하는가 / 이동하는가 / 경로만 참조하는가: 드래그 앤 드롭 시 앱 전용 라이브러리 폴더로 복사
- 원본 파일명 보존 여부: 메타데이터로 원본 파일명 별도 저장
- 저장 폴더 구조: `Library/Papers/{Year}/{FirstAuthor}_{Year}_{ShortTitle}.pdf`
- 동일 파일 재업로드 처리: DOI 또는 파일 해시가 같으면 중복 경고 후 사용자가 무시하고 추가 가능

### 5-2. 파일명 규칙

`작성:`
- 기본 규칙: `FirstAuthor_Year_Title.pdf`
- 대체 규칙: 저자/연도를 못 찾으면 `Title.pdf`
- 충돌 시 처리: 동일 이름이 이미 있으면 `_001`, `_002` 식으로 번호 부여
- 제목이 너무 길 때 처리: 공백 포함 80~100자 내외로 축약
- Windows 특수문자/경로 길이 처리 기준: `<>:"/\\|?*` 제거, 후행 마침표/공백 제거, 전체 경로 길이 안전 범위 내 유지

### 5-3. 사용자가 직접 수정 가능한 범위

`작성:`
- 파일명 수정 가능 여부: 가능
- 파일 위치 변경 가능 여부: MVP에서는 앱 내부 위치만 허용, 외부 arbitrary path 이동은 제외
- 자동 정리 후 수동 고정 기능 필요 여부: 필요. 사용자가 수정한 파일명은 재처리 시 덮어쓰지 않음

### 5-4. 삭제 및 휴지통 정책

`작성:`
- 논문 삭제 시 즉시 완전 삭제할지: 아니오. 먼저 휴지통으로 이동
- 휴지통에서 복원 가능 여부: 가능
- 휴지통에서 다시 삭제 시 완전 삭제할지: 예
- 완전 삭제 시 대상: DB 레코드, 로컬 PDF, figure asset, 연관 메모/하이라이트/요약 데이터

### 5-5. 백업 정책

`작성:`
- 백업 생성 주기: 30분 간격 자동 갱신
- 백업 범위: DB + 로컬 PDF 경로 기준 파일 + 설정 + 레이아웃 + 하이라이트/메모
- 백업 파일 portability: 다른 사용자의 앱에 넣으면 동일한 구조와 데이터가 복원되도록 설계
- 권장 백업 형태: import 가능한 단일 workspace backup package

### 5-6. 라이브러리 조직 방식

`작성:`
- 논문은 프로그램 내부 폴더로 분류할 수 있어야 함
- 폴더는 실제 디스크 폴더와 별개인 논리적 분류 구조로 관리
- 태그와 폴더는 동시에 사용 가능해야 함
- 하나의 논문은 여러 폴더에 들어갈 수 있게 할지: 예
- 폴더 계층 구조 지원 여부: 지원
- 대표 사용 예: `PSA`, `발표용`, `리뷰 예정`, `내 논문 근거`

## 6. 논문 카드 정의

### 6-1. 카드의 목적
카드는 무엇을 빠르게 보여주기 위한 화면인가?

`작성:`

> 논문 한 편이 무엇을 다루는지, 왜 중요한지, 내 연구와 어떤 관련이 있는지, 나중에 다시 봐야 하는지 여부를 짧은 시간 안에 판단하게 해주는 요약 화면이다.

### 6-2. 카드 생성에 필요한 기본 필드
LLM 없이도 카드가 먼저 만들어지기 위해 필요한 항목.

`작성:`
- 제목
- 저자
- 저널 / 학회
- 연도
- DOI
- 초록
- PDF 경로
- 읽음 상태
- 태그

### 6-3. 분석/요약 필드
이 항목들은 카드의 가치에는 중요하지만, 생성 시점에 반드시 채워질 필요는 없다.  
LLM이 초안을 만들거나 사용자가 직접 작성/수정하는 방식으로 채운다.

`작성:`
- 한 줄 요약
- 핵심 목적
- 핵심 방법
- 주요 결과
- 한계점
- 내 메모

### 6-4. 선택 필드
있으면 좋지만 필수는 아닌 항목.

`작성:`
- 왜 읽었는지
- 내 연구와의 관련성
- 발표 활용 가능 여부
- 다시 볼 이유
- 핵심 figure
- 실험 조건 요약
- 관련 논문 링크

### 6-5. 자동 생성 항목 vs 사용자 작성 항목

`작성:`
- 자동 생성: 제목, 저자, 저널, 연도, DOI, 초록, 섹션, chunk, figure/caption
- LLM 초안 생성: 한 줄 요약, 핵심 목적, 핵심 방법, 주요 결과, 한계점
- 사용자 작성: 개인 메모, 태그, 읽음 상태, 중요 표시
- 자동 생성 후 사용자 수정 가능 항목: 메타데이터, 요약, 분석 필드 전반

## 7. 메모 체계

### 7-1. 메모의 역할
메모는 자유 기록인지, 구조화된 연구 자산인지 정의한다.

`작성:`

> 메모는 단순 감상 기록이 아니라, 나중에 재검색과 근거 재사용이 가능하도록 논문·섹션·figure와 연결되는 구조화된 연구 자산이다.

### 7-2. 메모 타입
처음부터 타입을 나누면 나중에 검색과 ontology 확장에 유리하다.

`작성:`
| note_type | 설명 | 필수 여부 | 예시 |
|---|---|---|---|
| summary_note | 논문 전체를 한두 문장으로 요약 | 선택 | "PSA 조건 변화가 purity/recovery trade-off에 미치는 영향 정리" |
| relevance_note | 내 연구와 어떤 관련이 있는지 기록 | 선택 | "내 layered bed 실험 설계와 직접 관련" |
| presentation_note | 발표 자료에 쓸 수 있는 포인트 기록 | 선택 | "Figure 4는 발표용 도식으로 적합" |
| result_note | 중요한 결과나 수치를 기록 | 선택 | "압력 증가 시 recovery 감소 경향" |
| followup_note | 나중에 다시 봐야 하는 이유 기록 | 선택 | "조건표를 실험 설계 참고용으로 재확인 필요" |
| figure_note | 특정 figure에 대한 해석/활용 기록 | 선택 | "breakthrough curve 비교 시 이 그림 우선 참고" |
| question_note | 읽다가 든 의문점이나 검증 포인트 | 선택 | "CO2 영향이 bed 구조 때문인지 feed 조건 때문인지 불명확" |
| custom | 위 분류에 딱 맞지 않는 사용자 자유 메모 | 선택 | "후속 아이디어 메모" |

### 7-3. 메모 입력 방식

`작성:`
- 자유 텍스트만 둘지 / 템플릿형 입력도 줄지: 둘 다 제공. 기본은 자유 텍스트, 주요 note_type은 템플릿 버튼 제공
- 논문 단위 메모와 section/chunk/figure 단위 메모를 모두 둘지: 모두 둠
- 태그를 메모에도 붙일지: 붙일 수 있게 함
- 폴더와 메모의 관계: 메모 자체는 폴더에 넣지 않고, 논문을 통해 간접적으로 분류

## 8. 읽음 상태 및 우선순위

### 8-1. 읽음 상태 정의
상태는 가능한 한 적고 명확해야 한다.

`작성:`
| 상태 | 의미 | 사용 기준 |
|---|---|---|
| 읽기 전 | 저장만 했고 아직 읽지 않음 | 업로드 직후 기본값 |
| 읽는 중 | 일부 읽었거나 메모 작성 중 | 초록만 읽었거나 중간 단계 |
| 읽음 | 핵심 내용을 파악했고 카드/메모 정리가 끝남 | 최소 1개 이상 메모 또는 요약 완료 |
| 중요 논문 | 자주 다시 볼 핵심 논문 | 연구 주제와 직접 관련 |
| 다시 볼 논문 | 아직 소화 안 됐지만 재검토 필요 | 보류 또는 후속 검토 대상 |
| 제외 | 중복이거나 내 연구와 무관해 실질적으로 보관만 함 | 삭제 대신 보관 |

### 8-2. 중요도/즐겨찾기 정책

`작성:`
- 중요 논문은 상태인지 별도 플래그인지: 별도 플래그 권장
- 발표용 스크랩 후보는 별도 플래그인지: 별도 플래그 권장
- 다시 볼 논문과 중요 논문은 중복 가능한지: 가능

## 10. Chunk 정책

### 10-1. Chunk 목적
chunk는 검색과 provenance를 위한 단위인지, 요약 단위까지 겸하는지 적는다.

`작성:`

> chunk는 검색, 근거 제시, provenance 추적을 위한 기본 단위이며, 필요하면 요약과 claim 비교의 입력 단위로도 사용한다.

### 10-2. Chunk 생성 기준

`작성:`
- 문단 단위 / 문장 묶음 / 토큰 기준: 기본은 문단 단위, 긴 문단은 길이 기준으로 추가 분할
- chunk 최대 길이: 대략 500~800 tokens 또는 문단 1~3개
- overlap 여부: 긴 chunk를 나눌 때만 소량 overlap 허용
- section 정보 포함 여부: 포함
- page 정보 포함 여부: 포함
- 순서 정보 포함 여부: 포함

### 10-3. 나중에 꼭 필요할 provenance 정보

`작성:`
- section_name
- page
- chunk_order
- source_text_span 필요 여부: 가능하면 저장
- figure/table 연결 필요 여부: 필요
- 기타: 추출 방식, 생성 시각, 생성 주체(system/user/llm)

## 12. 검색과 재검색 규칙

### 12-1. 지금 단계에서 필요한 검색
RAG 전에도 필요한 기본 검색을 적는다.

`작성:`
- 제목 검색: 지원
- 저자 검색: 지원
- DOI 검색: 지원
- 폴더 검색/필터: 지원
- 태그 검색: 지원
- 내 메모 검색: 지원
- section/chunk 본문 검색: 지원
- figure caption 검색: 지원
- semantic/vector 검색: MVP에서는 구조만 준비하고 우선순위는 낮춤

### 12-2. 근거 검색의 기본 원칙

`작성:`
- Results / Discussion / Conclusion 우선 여부: 우선
- Introduction은 배경 설명으로만 취급할지: 기본적으로 그렇다
- 근거 없음이면 "없음" 반환할지: 그렇다
- 억지 매칭 금지 기준: 키워드만 비슷하고 결과/조건/방향성이 다르면 매칭하지 않음

### 12-3. 검색 결과에 반드시 보여줄 정보

`작성:`
- 논문명
- section
- page
- 근거 문단
- figure/table 연결 여부
- 매칭 수준
- 짧은 설명
- 관련 조건 또는 실험 맥락

### 12-4. 매칭 수준 정의

`작성:`
| 수준 | 의미 | 사용 기준 |
|---|---|---|
| 직접 일치 | 사용자의 주장과 논문 결과가 거의 같은 의미 | 방향성, 대상, 결과, 조건이 모두 유사 |
| 부분 일치 | 일부만 맞고 조건이나 표현 범위가 다름 | 핵심 현상은 비슷하지만 조건 차이 존재 |
| 확장 참고 | 직접 근거는 아니지만 주변 맥락으로 참고 가능 | 동일 주제/기작/배경 설명 수준 |
| 불일치 | 의미상 지지하지 않거나 반대됨 | 결과 방향이 다르거나 근거가 없음 |

## 13. LLM 사용 범위

### 13-1. LLM이 맡는 일

`작성:`
- 한 줄 요약
- 목적/방법/결과/한계점 정리
- claim 해석
- 검색 후보 근거성 검증
- 조건 정리
- section/chunk 기반 짧은 설명 생성
- vector 생성 이후 자동 요약 생성

### 13-2. LLM이 맡지 않는 일

`작성:`
- PDF 텍스트 추출
- 섹션 분리
- DOI 탐지
- figure 식별
- 파일 구조 관리
- 검색 후보 1차 수집
- 중복 파일 판별의 1차 기준

### 13-3. 사용자에게 언제 LLM 결과를 보여줄지

`작성:`
- 업로드 직후 자동 생성할지: 메타데이터와 본문 저장 후, vector 생성이 끝나면 자동으로 백그라운드 생성
- 버튼 클릭 시 생성할지: 가능. 재생성 버튼 제공
- 결과 수정 가능 여부: 가능
- LLM 결과와 사용자 메모를 구분해서 보여줄지: 반드시 구분

### 13-4. LLM 실행 순서

`작성:`
- 1차: PDF 업로드 및 텍스트/OCR 추출
- 2차: 섹션 분리, chunk 생성, provenance 저장
- 3차: vector 생성
- 4차: LLM 요약 생성
- 원칙: 요약은 retrieval-ready 상태 이후에 생성한다

## 15. 품질 기준과 성공 기준

### 15-1. 품질 기준
정확도보다 사용자 체감 관점으로 적어도 좋다.

`작성:`
- 메타데이터 추출이 어느 정도 정확해야 하는가: 제목/저자/연도 수준에서 체감상 대부분 바로 쓸 수 있어야 함
- 섹션 분리가 어느 정도 안정적이어야 하는가: 주요 섹션(Abstract, Methods, Results, Discussion) 구분이 대다수 논문에서 동작해야 함
- figure/caption 연결이 어느 정도 정확해야 하는가: figure 번호와 caption 연결은 실사용 가능 수준이어야 함
- 검색 결과에서 허위 근거를 어느 정도까지 허용하지 않을 것인가: Introduction 일반론만으로 직접 근거라고 보여주면 안 됨
- OCR 품질은 어느 정도여야 하는가: 스캔 PDF도 최소 제목/초록/주요 섹션 수준의 복구가 가능해야 함

### 15-2. MVP 성공 기준
숫자나 검증 가능한 형태로 적는다.

`작성:`
- 논문 업로드 후 카드 생성 시간: 1분 이내
- 재검색 속도: 일반 검색 3초 이내, 근거 확인 포함 10초 이내
- 사용자가 다시 찾을 수 있어야 하는 정보: 논문 제목, 특정 결과, figure, 개인 메모, 태그
- 몇 편 정도의 논문까지 실용적이어야 하는가: 최소 수백 편 규모
- 사용 후 어떤 일이 더 쉬워져야 하는가: "예전에 읽은 논문 다시 찾기"와 "발표용 figure 재확인"이 훨씬 빨라져야 함
- 자동 백업은 얼마나 안정적이어야 하는가: 30분 간격으로 갱신되며 다른 설치 환경에서도 복원 가능해야 함

### 15-3. 실패 기준
어떤 상태면 MVP가 실패한 것으로 볼지 적는다.

`작성:`
- 업로드는 되지만 정리 결과를 사용자가 신뢰하지 못하는 경우
- 다시 찾기 기능이 파일 탐색기와 큰 차이가 없는 경우
- 자동 추출 오류를 사용자가 쉽게 수정할 수 없는 경우

### 15-4. 계정 및 인증 정책

`작성:`
- 기본 인증 방식: 일반 계정 생성
- 소셜 로그인: Google 로그인 지원
- 사용자별로 분리되어야 하는 데이터: 하이라이트 preset, 메모, 레이아웃, 개인 태그
