# Redou 개발용 서브에이전트 추천

> 이 프로젝트의 특성(Electron + React + Supabase + AI/OCR 파이프라인)에 맞춰  
> Claude Code에서 활용할 수 있는 서브에이전트 전략을 추천합니다.

---

## 1. 지금 바로 쓸 수 있는 내장 에이전트

### Explore (코드베이스 탐색)
- **용도**: 파일 검색, 코드 패턴 검색, 아키텍처 파악
- **언제 쓰면 좋은가**:
  - "이 IPC 채널 어디서 쓰이지?" 같은 사용처 추적
  - "figure 관련 코드 전체 흐름 파악" 같은 넓은 탐색
  - 새 기능 추가 전 관련 코드 파악
- **예시**: `"CHAT_SEND_MESSAGE 채널의 전체 흐름을 추적해줘 — main.mjs 핸들러부터 프론트엔드 호출까지"`

### Plan (구현 계획 설계)
- **용도**: 기능 추가/리팩토링 전 아키텍처 수준 계획
- **언제 쓰면 좋은가**:
  - 새 기능(태그 자동 분류, 논문 추천 등) 구현 전 설계
  - DB 스키마 변경이 필요한 작업의 영향도 분석
  - 여러 모듈에 걸친 변경 시 순서/의존성 정리
- **예시**: `"논문 간 인용 네트워크 시각화 기능을 추가하려면 어떤 순서로 작업해야 할지 계획해줘"`

### frontend-design (프론트엔드 UI)
- **용도**: 프로덕션급 UI 컴포넌트/페이지 생성
- **언제 쓰면 좋은가**:
  - 새 뷰 컴포넌트 작성 (디자인 퀄리티가 중요할 때)
  - 기존 UI 리디자인
  - 복잡한 인터랙션 컴포넌트 (드래그 앤 드롭, 차트, 타임라인 등)
- **예시**: `"논문 인용 네트워크를 인터랙티브 그래프로 시각화하는 컴포넌트를 만들어줘"`

### codex:rescue (2차 진단/구현)
- **용도**: 막혔을 때 다른 관점에서 조사, 대안 구현
- **언제 쓰면 좋은가**:
  - 버그 원인을 못 찾겠을 때
  - 성능 최적화 방향이 불분명할 때
  - 이미 시도한 접근이 실패했을 때 다른 관점 필요

## 2. 프로젝트 특성에 따른 활용 전략

### 시나리오별 추천 조합

#### A. 새 기능 추가 (예: 논문 추천 시스템)
```
1. Plan 에이전트      → 아키텍처 설계, DB 스키마, 파이프라인 흐름
2. 직접 작업          → 마이그레이션 SQL, Electron 모듈, IPC 채널
3. frontend-design    → UI 컴포넌트 (추천 카드, 리스트 등)
4. 직접 테스트/디버깅  → 통합 확인
```

#### B. 기존 기능 버그 수정
```
1. Explore 에이전트   → 관련 코드 흐름 추적
2. 직접 수정          → 버그 원인 파악 후 수정
3. (막힐 경우) codex:rescue → 다른 관점에서 진단
```

#### C. 대규모 리팩토링
```
1. Explore 에이전트   → 영향 범위 파악 (병렬로 여러 검색)
2. Plan 에이전트      → 단계별 리팩토링 계획
3. 직접 작업          → 단계별 실행
```

#### D. PDF 파이프라인 개선
```
1. Explore 에이전트   → pdf-heuristics/ocr-extraction/mineru-client 흐름 파악
2. Plan 에이전트      → 파이프라인 변경 계획 (버전 범프 포함)
3. 직접 작업          → 모듈 수정 + CURRENT_EXTRACTION_VERSION 증가
```

## 3. 병렬 작업 패턴

독립적인 작업은 에이전트를 병렬로 실행하면 효율적입니다:

```
병렬 가능한 조합:
├── Explore: "프론트엔드에서 figure 데이터 조회 흐름"  (백그라운드)
├── Explore: "Electron에서 figure 처리 파이프라인"      (백그라운드)
└── 직접: DB 마이그레이션 SQL 작성
```

## 4. 사용 팁

### 에이전트를 쓰는 게 유리한 경우
- 3개 이상의 파일을 넘나드는 코드 추적
- 프로젝트 전반에 걸친 패턴 검색
- 복잡한 UI 컴포넌트 신규 작성
- 아키텍처 수준 설계가 필요한 작업

### 에이전트 없이 직접 하는 게 나은 경우
- 특정 파일 1-2개 수정
- 단순 타입 변경, 문자열 수정
- 이미 위치를 아는 코드 수정
- `Glob`이나 `Grep`으로 바로 찾을 수 있는 검색

### 에이전트에게 좋은 프롬프트 작성법
```
❌ "figure 관련 코드 찾아줘"
✅ "pdf-heuristics.mjs에서 figure를 감지한 후, 그 결과가 figures 테이블에 
   저장되기까지의 전체 흐름을 추적해줘. main.mjs의 어떤 함수가 오케스트레이션하는지,
   프론트엔드에서는 어떤 쿼리 키로 조회하는지 포함해서 정리해줘."
```

## 5. 이 프로젝트에서 특히 유용할 추가 도구들

| 도구/기능 | 용도 | 비고 |
|-----------|------|------|
| `/simplify` | 작성한 코드 품질 리뷰 | 코드 작성 후 자동 리뷰 |
| `/commit` | 커밋 메시지 자동 생성 | 변경 분석 후 커밋 |
| `schedule` | 정기 작업 예약 | CI/CD 자동화 등 |
| `WebSearch` | 최신 라이브러리/API 문서 검색 | pdfjs, Supabase 최신 API 확인 |
